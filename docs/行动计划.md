# 关键保命策略（必看）
## 1. 数据库风险控制
- 全程使用 PostgreSQL（Neon），避免临时切换数据库导致的 schema 漂移；
- 数据库仅做「最小化读写」：用户查询、订单创建、交易落库、订阅生效，不做复杂分析；
- 本地测试通过后再部署，避免线上数据库卡点。

## 2. Stripe卡点应对
- 彻底放弃真实Stripe调用，用「模拟接口+前端样式」完成演示；
- 保留Stripe SDK和接口参数，后续注册成功后可快速切换为真实接口。

## 3. 工作量控制
- 每天只做「底线任务」，完成即停，不额外加功能；
- 数据库字段只保留主链路核心字段，非核心功能（复杂优惠、对账、风控）延后；
- 密码使用哈希存储（不再使用明文）。

## 4. 时间分配
- 认证+数据库（Day1+Day2上午）：合计1.5天；
- 支付（Day2下午+Day3）：合计1.5天；
- 整合+部署（Day4+Day5）：合计2天。
